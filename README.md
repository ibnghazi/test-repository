# 🚀 [프로젝트명] Git 협업 가이드라인

우리 팀의 평화로운 개발과 칼퇴를 위한 Git 작업 규칙입니다.
작업하기 전에 꼭 한 번 읽어주세요!

## 1. 기본 원칙 (The Golden Rules)
1. **작업 전 무조건 Pull:** 출근하면 `git pull`부터 합니다.
2. **Main 브랜치 금지:** `main` 브랜치에서 직접 작업하지 않습니다. 반드시 내 브랜치를 따서 작업합니다.
3. **Commit 메시지는 명확하게:** "수정함" (X) -> "로그인 버튼 색상 변경" (O)

## 2. 작업 순서 (Workflow) - 이것만 지켜주세요!

**STEP 1. 최신 코드 받기 (출근/작업 시작 전)**
터미널을 열고 무조건 아래 명령어를 입력합니다.
git pull origin main

**STEP 2. 내 작업 공간 만들기**
내가 맡은 기능 이름으로 브랜치를 생성하고 이동합니다.
git checkout -b [기능이름-본인이름]
(예: git checkout -b login-chulsu)

**STEP 3. 작업 및 저장 (Commit)**
열심히 코딩한 후, 저장합니다.
git add .
git commit -m "[태그] 작업 내용 요약"
(예: git commit -m "Feat: 회원가입 페이지 UI 완성")

**STEP 4. 업로드 (Push)**
내 작업 공간을 GitHub로 보냅니다.
git push origin [내-브랜치-이름]
(예: git push origin login-chulsu)

**STEP 5. 검토 요청 (PR)**
GitHub 웹사이트로 이동해서 **[Pull Request]** 버튼을 누르고 PR을 생성합니다.
팀원들에게 리뷰를 요청합니다.

---

## 3. 🚨 비상 대처 매뉴얼 (충돌 발생 시)

`git pull`이나 `merge` 중에 **CONFLICT** 에러가 떴다면? 당황하지 마세요!

1. 에러 메시지에 뜬 파일을 엽니다. (VS Code 등 에디터 사용)
2. `<<<<<<<`, `=======`, `>>>>>>>` 기호가 보일 겁니다.
3. **팀장님**이나 코드를 작성한 동료와 상의해서 남길 코드를 정하고, 나머지 코드와 특수 기호들을 다 지웁니다.
4. 다시 `git add .` -> `git commit` -> `git push` 하면 해결!

## 4. 자주 쓰는 명령어 요약표
| 명령어 | 설명 |
| :--- | :--- |
| `git pull origin main` | 원격 저장소의 최신 내용을 가져옵니다. |
| `git checkout -b [이름]` | 새로운 브랜치를 만들고 그곳으로 이동합니다. |
| `git status` | 현재 파일들의 상태를 확인합니다. (수시로 확인 추천) |
| `git add .` | 변경된 모든 파일을 저장 대기 상태로 만듭니다. |
| `git commit -m "메시지"` | 대기 상태인 파일들을 역사(버전)로 기록합니다. |
| `git push origin [이름]` | 내 브랜치를 GitHub에 올립니다. |
